# Operating Model

## Goal

This document defines the behavioral boundaries for `create-report` during
requirement clarification, spec validation, downstream skill generation, and
standalone packaging.

## Required Stages

`create-report` must work in this order:

1. intake
2. extraction
3. clarification
4. normalization
5. validation
6. generation

Do not jump from intake directly to generation.

## Clarification Rules

Must clarify when:

- report readers or decision goals are unclear
- a section cannot be mapped to a dataset
- a metric name exists but its source field or formula is unknown
- the time window is missing or conflicts
- conclusions are required but the supporting rules are undefined
- classification logic depends on missing attachments

Safe defaults are allowed only when:

- the final format is not specified and `html` is a reasonable default
- a spec field is optional and does not affect execution
- wording differs but business semantics do not

Never guess:

- metric definitions
- segment rules
- chart field choices
- cross-dataset joins
- unsupported causal claims

## Blocking Conditions

Do not generate a downstream skill when any of these are unresolved:

- report audience
- decision goals
- section list
- section-to-dataset mapping
- metric definitions
- time definitions
- output contract
- downstream skill name

For `runnable` downstream skills, also block when:

- required data contracts do not have minimal real sample files
- sample file names do not match the declared `file_name_pattern`
- samples are missing required execution fields
- samples do not cover at least two comparable periods
- ratio metrics do not declare whether to use raw values or recomputation

## Allowed Inference Scope

Allowed:

- rewrite repetitive prose into contract language
- merge duplicated business statements
- infer stable `section_id` values from explicit section titles
- generalize specific file examples into file-name patterns

Not allowed:

- invent missing business logic
- invent chart types not implied by the requirement or template
- promote optional data into required data without evidence
- infer KPI formulas from field names alone

## Output Rules

All downstream report skills generated by `create-report` must target HTML.

The minimal HTML contract must include:

- a stable title block
- period metadata
- fixed section order matching the spec
- chart and table regions where required
- section-level conclusions
- source-data or citation notes when needed

If a section includes tables or structured conclusions, the spec must define:

- fixed table column names
- which columns require WoW display
- which fields must explicitly say up / down / flat
- which sections use charts, tables, or both

Section titles alone are not enough to justify a `runnable` output.

## Downstream Skill Levels

### `documentation-only`

Use when the requirement can support a normalized spec, but real samples or
execution details are still missing.

Allowed outputs:

- normalized spec
- downstream skill package structure
- HTML contract
- input inventory
- documentation-only script stubs

Restrictions:

- do not claim the skill is runnable
- do not present placeholder HTML as runnable proof

### `runnable`

Use only when:

- the normalized spec is complete
- real sample files exist
- the output contract defines table schemas, narrative schemas, and WoW rules
- execution has been verified with a real run

Additional requirements:

- real execution script
- dependency declaration
- at least one regression test
- one HTML output generated from real samples

## Standalone Publishability

If a downstream skill is intended for independent publication, it must also:

- be self-contained
- avoid absolute paths
- avoid references to upstream repo-only documents
- embed the input contract within the package
- avoid caches, local artifacts, and temporary files

If those checks fail, treat the package as repo-local only.

## Post-Generation Verification

For `runnable` outputs, verify more than file existence:

- `run-report.sh` or equivalent actually runs
- output HTML is not a placeholder
- key sections required by the spec exist
- required charts or equivalent structures exist
- required tables match the declared schema
- required direction words appear in conclusions when the spec demands them
