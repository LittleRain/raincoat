# Reports 周报系统设计

## 概要

Raincoat 首期将 `trading-weekly` 落成一个“脚本托管型单场景工具”。

首期目标不是做完整多场景 BI 平台，而是先形成最小可用闭环：

- 在页面展示周报场景固定名称和说明
- 上传本次周报所需文件并触发脚本运行
- 保存运行记录、脚本版本、输入文件清单和产物路径
- 可查看历史记录，并打开对应生成的 HTML

在这个闭环上，平台同时为两类后续扩展预留结构：

- 更新已有周报脚本逻辑
- 添加新的周报场景，并支持为场景定义固定名称

## 设计原则

- 脚本逻辑优先复用，`trading-weekly` 不重写分析逻辑
- 平台只补托管能力，不侵入脚本内部实现
- 场景通过代码和 manifest 注册，避免首期引入后台配置系统
- 历史运行结果必须和当时脚本版本绑定，保证可追溯

## 模型约定

### 场景

场景由 `tools/reports/manifests/*.yaml` 注册，至少包含：

- `id`
- 固定显示名称
- 描述
- 路由前缀
- 默认激活的 `releaseId`

场景名称由开发手动配置，运行时不支持每次单独填写标题。

### 脚本版本

每个场景存在一个或多个 release。平台运行时需要固化本次使用的：

- `releaseId`
- `version`
- `scriptPath`

这样在“更新已有周报逻辑”后，历史运行记录仍然能说明它是基于哪个脚本版本生成的。

### 运行记录

一次执行对应一条运行记录，至少包含：

- `runId`
- `scenarioId`
- `scenarioName`
- `releaseId/version`
- `status`
- `createdAt` / `startedAt` / `finishedAt`
- 输入文件清单
- 产物路径
- 失败摘要和日志路径

## 运行流程

### 页面层

- `/reports` 展示场景列表
- `/reports/:scenarioId` 展示场景详情、上传入口和最近运行记录

### API 层

- 查询场景列表和场景详情
- 接收一次执行请求和上传文件
- 创建运行目录和运行记录
- 提供运行记录查询和 HTML 预览访问入口

### Worker 层

- 根据场景找到当前激活 release
- 将上传文件落到本次运行的 `input-dir`
- 调用脚本并传入 `--input-dir --output-dir --run-id`
- 收集 `report.html` 和日志文件
- 回写成功或失败状态

## 脚本 Contract

所有周报脚本统一遵守最小 contract：

- 输入通过目录方式提供
- 输出目录内至少生成 `report.html`
- 平台通过脚本 manifest 发现入口和产物约定

`trading-weekly` 首期直接复用现有 Python 原型，只允许调整输入输出适配，不改动核心周报分析逻辑。

## 后续扩展

### 更新已有周报脚本逻辑

- 新逻辑以新 release 形式接入
- 默认激活 release 可在场景 manifest 中切换
- 历史运行记录继续引用旧版本信息和旧 HTML 产物

### 添加新的周报场景

- 新增脚本目录
- 新增场景 manifest
- 在 manifest 中定义固定场景名称和描述
- 复用同一套上传、运行、记录、预览能力

## 首期不做

- 后台动态创建场景
- 每次运行自定义标题
- 通用调度系统或消息队列
- 数据库存储
- 多机执行
