# 创建周报 Skill 验证规则

## 目的

本文档定义 `create-report` 的验证层和发布门槛。

因为这个 skill 要开放给公司内部使用，所以它必须在原始材料缺失或冲突时，
稳定地“安全失败”，而不是生成错误结果。

## 验证层

### 1. Spec 完整性校验

检查标准化 spec 是否包含所有必需字段。

核心检查项：

- `report_goal` 完整
- `report_outline` 至少有一个栏目
- 每个栏目都声明了指标、维度和输出形式
- `data_contracts` 覆盖所有必需栏目
- `output_contract.format` 为 `html`

### 2. Spec 一致性校验

检查标准化 spec 内部是否自洽。

核心检查项：

- 栏目指标至少能在一个数据合同中找到
- 时间定义在不同数据集之间没有冲突
- 术语规则引用了真实字段
- 输出栏目顺序和声明的栏目集合一致

### 3. 生成就绪校验

检查 spec 是否已经足以生成下游 skill 包。

核心检查项：

- `skill_generation_contract` 完整
- 所需文件、资产和样例已声明
- 没有尚未解决的追问项
- 没有触发禁止推断规则

### 4. HTML 输出合同校验

检查下游 skill 是否能稳定输出约定的 HTML 结构。

核心检查项：

- 必须有标题区和周期信息区
- 栏目顺序稳定
- 图表和表格区已声明
- 需要时有结论区
- 需要时有数据说明或引用说明

## 样例集

### 黄金样例

- [圈子业务周报需求.md](/Users/raincai/Documents/GitHub/raincoat/docs/skills/圈子业务周报需求.md)

预期行为：

- 能稳定映射为标准化 spec
- 在补齐必要信息后，通过完整性和一致性校验
- 能生成合法的 `report-circle-weekly` skill 包

### 缺失信息样例

至少维护以下负向样例：

- 缺时间定义
- 缺指标定义
- 缺栏目与数据源映射
- 缺输出合同

预期行为：

- 阻塞生成
- 返回缺口清单和补充要求

### 冲突样例

至少维护以下冲突样例：

- 同一指标在不同材料中定义不一致
- 不同数据集使用不同时间口径
- 分类规则依赖冲突字段

预期行为：

- 阻塞生成
- 明确指出冲突位置

### 跨业务域样例

后续至少补充两个非圈子业务域样例。

预期行为：

- 复用同一套 spec 合同
- 不出现圈子业务特有假设泄漏

### 调整回归样例

每当首次生成后的下游 skill 因为“结果不符合预期”而发生定向调整时，
都应该补充一个调整回归样例。

至少记录：

- 原始偏差描述
- 偏差归属层级
  - 原始需求
  - 标准化 spec
  - `create-report` 模板
  - 下游周报 skill
- 修改内容
- 回归验证结果

## 发布门槛

在开放给更多内部团队前，至少满足以下条件：

- 黄金样例能完整跑通
- 所有缺失样例都能安全失败
- 所有冲突样例都能安全失败
- 生成的 HTML 结构稳定
- 其他内部同学不依赖隐藏上下文也能读懂生成产物

## 持续维护规则

每次真实使用中的失败或评审争议，都应该回流为以下至少一项：

- 新的 validation case
- 更强的阻塞规则
- 更清晰的 spec 字段定义
- 一条明确的“生成后定向调整”记录

验证规则不是附属说明，而是产品的安全边界。
