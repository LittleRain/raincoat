# 创建周报 Skill 生产器 PRD

## 背景与目标

`create-report` 是一个面向公司内部的周报 skill 生产器。

它不是单一的“写周报”skill，而是负责接收业务方提供的半结构化材料，
先把需求澄清清楚，再沉淀成稳定的中间规格，最后生成一个具体业务域的
下游周报 skill。下游周报 skill 的最终输出统一为 HTML 周报。

这个产品存在的原因是：业务方给周报需求时，往往会混合提供以下材料：

- 业务背景和领域名词
- 历史周报模板或截图
- 数据源说明和报表字段
- 文件命名习惯
- 口头约定但未写清楚的分析预期

`create-report` 的职责，就是把这类不稳定输入转成稳定、可复用、可校验、
可回放的生成流程。

## 目标用户

### 主要用户

- 知道自己想要什么周报，但描述仍是半结构化的业务方
- 帮业务团队收集周报需求的分析或运营同学

### 次级用户

- 维护周报 skill 的内部研发或搭建同学
- 审查生成结果是否安全可用的评审者

## 核心流程

1. 业务方提交原始材料
   - 可以是一份需求 markdown、历史周报、数据源说明、字段定义、
     附件说明等
2. `create-report` 解析输入并抽取已知事实
   - 包括读者、决策目标、栏目结构、数据文件、指标定义、时间周期、
     分类规则、输出形式等
3. `create-report` 识别需求缺口和冲突
   - 包括缺字段、指标定义不清、栏目无数据支撑、规则互相冲突等
4. `create-report` 进行聚焦追问
   - 只追问“生成所必需”的信息，不做开放式咨询
5. `create-report` 产出标准化周报 spec
   - 该 spec 是业务意图和下游 skill 生成之间的唯一正式边界
6. `create-report` 校验 spec 是否可以执行
   - 若字段缺失或规则冲突，则停止生成并返回缺口报告
7. `create-report` 生成具体的下游周报 skill
   - 输出 `report-<domain>` skill 包，包含说明、元数据、模板、样例等
8. 下游 skill 消费真实周报数据并输出 HTML

## 关键能力

### 1. 需求抽取

能够从半结构化业务材料中抽取稳定信息，包括：

- 业务目标和读者
- 名词定义和分类口径
- 栏目结构
- 数据文件合同和字段预期
- 分析规则和阈值
- HTML 输出要求

### 2. 缺口识别与追问

能够发现需求不完整、定义模糊或内部冲突的地方。

典型情况包括：

- 某个栏目存在，但没有可映射的数据源
- 提到了指标，但没有字段来源或计算口径
- 要求同比/环比，但时间定义不清
- 要求归因，但没有允许使用的证据边界

### 3. 规格标准化

能够把原始业务描述重组为统一的 spec，而不是保留为松散文档。
下游 skill 只能消费标准化 spec，不能直接消费原始业务 prose。

### 4. 保守校验

当稳定性规则不满足时，必须拒绝生成。缺字段时要输出缺口报告，
而不是猜测性地补全结果。

### 5. Skill 组装

能够根据标准化 spec 生成符合 Raincoat 约定的 `report-<domain>` skill，
并且保证其他内部同学不依赖隐性上下文也能理解和使用。

### 6. 输出合同固化

能够把业务方要求的输出形式固化进下游 skill，重点包括：

- 最终输出为 HTML
- 栏目顺序稳定
- 图表和表格区块明确
- 结论区和建议区可预期
- 需要时包含数据说明或引用说明

## 成功标准

`create-report` 成功的定义是：

- 能把
  [圈子业务周报需求.md](/Users/raincai/Documents/GitHub/raincoat/docs/skills/圈子业务周报需求.md)
  稳定映射为标准化 spec，且不丢失关键业务语义
- 必填字段缺失时会停止生成，而不是输出“看似完整”的错误产物
- 生成出的下游 skill 可以被其他内部同学审阅和复用
- 下游 skill 输出 HTML 的栏目结构是稳定、可预期的
- 整个生成过程可通过 spec 和校验结果进行复盘

## 非目标

当前阶段不追求：

- 搭建一个全自动、全业务域的 BI 平台
- 根据弱提示自动脑补业务逻辑
- 替代业务定义上的人工 review
- 支持 HTML 以外的任意输出形式
- 在质量门槛未跑通前优先做自动化深度

## 稳定性要求

因为这个 skill 要开放给公司内部使用，所以稳定性优先于自动化程度。

### 强制规则

- spec 必填字段缺失时，禁止生成
- 不允许脑补行业口径、指标含义、图表字段
- 没有声明数据合同支撑的分析要求，禁止生成到下游 skill 中
- HTML 输出结构必须与通过校验的合同一致

### 必须具备的校验层

- 完整性校验
- 一致性校验
- 可执行性校验
- HTML 输出合同校验

### 必须产出的审阅材料

每次生成至少要留下：

- 一份标准化 spec 或 spec 摘要
- 一份校验结果
- 若被阻塞，则输出一份明确的缺口列表
