# 创建周报 Skill 运行模型

## 目的

本文档定义 `create-report` 在需求澄清、spec 校验、下游 skill 生成过程中的
行为边界。

这个运行模型的目标是：让 `create-report` 可以被公司其他团队复用，
而不是只在当前上下文里偶然可用。

## 工作阶段

`create-report` 必须严格按以下 6 个阶段工作：

1. intake
2. extraction
3. clarification
4. normalization
5. validation
6. generation

禁止从 intake 直接跳到 generation。

## 追问策略

### 以下情况必须追问

- 周报读者或决策目标不清晰
- 某个栏目无法映射到数据集
- 指标有名字，但字段来源或公式未知
- 时间窗口缺失、相互冲突或只有模糊描述
- 需要输出结论，但对应规则没有定义
- 分类逻辑依赖附件，但附件没有说明或未提供

### 以下情况允许使用安全默认值

- 未明确写最终格式时，使用产品默认值 `html`
- 某字段在 spec 中是可选字段，且不影响可执行性
- 只是措辞差异，不影响业务语义

### 以下情况禁止猜测

- 指标定义
- 分类或分层口径
- 图表字段选择
- 跨数据集 join 逻辑
- 没有证据支持的因果解释或归因

## 阻塞条件

当以下任一字段缺失或未解决时，禁止生成下游 skill：

- 周报读者
- 决策目标
- 栏目清单
- 栏目到数据源的映射
- 指标定义
- 时间定义
- 输出合同
- 下游 skill 名称

当以下冲突未解决时，也必须阻塞：

- 指标名称或口径冲突
- 周期定义冲突
- 分类规则冲突
- 文件与栏目覆盖关系冲突

## 允许的推断范围

`create-report` 只允许做轻量格式层推断。

允许：

- 把重复描述整理成合同语言
- 合并重复业务表述
- 根据明确栏目标题推断稳定的 `section_id`
- 将具体文件示例提升为文件名 pattern

不允许：

- 推断缺失业务逻辑
- 发明未被需求要求或模板暗示的图表类型
- 把可选数据强行提升为必填数据
- 根据字段名称猜 KPI 公式

## HTML 输出规则

所有由 `create-report` 生成的下游周报 skill，最终输出必须是 HTML。

最小 HTML 合同包括：

- 稳定的标题区
- 报告周期信息区
- 与 spec 一致的固定栏目顺序
- 图表和表格占位或渲染区
- 栏目级结论区
- 需要时包含数据说明或引用说明区

同一次生成中，禁止输出多种相互竞争的格式合同。

## 失败处理

当 `create-report` 无法安全继续时，应返回：

- 简洁的失败原因
- 尚未解决的字段或冲突
- 失败所处阶段
- 继续所需的补充信息

优先选择显式失败，而不是部分生成。

## 恢复策略

当用户补充了新信息后，`create-report` 应：

1. 重新打开上一版标准化 spec 草稿
2. 只修补受影响的部分
3. 重新执行 validation
4. 仅在所有门槛通过后继续 generation

## 审阅产物要求

每次成功生成，都必须留下：

- 一份标准化 spec
- 一份校验结果或校验摘要
- 一份下游 skill 包生成记录

这样可以支持审计、回放和后续交接。

## 首次生成后的定向调整

`create-report` 首次生成出的下游周报 skill 如果“不符合预期”，默认不应直接
推倒重生，而应先进入“定向调整”流程。

### 允许触发定向调整的情况

- 栏目顺序不符合业务预期
- 某个栏目缺少必要表格、图表或结论区
- 文件匹配规则过宽或过窄
- 某些字段映射、阈值规则、分类口径与业务要求不一致
- HTML 结构可以运行，但表现不满足业务方要求

### 定向调整流程

1. 记录偏差
   - 明确是哪一栏、哪条口径、哪种输入文件或哪种输出结构不符合预期
2. 判断偏差层级
   - 如果是业务需求没写清，先修原始需求或标准化 spec
   - 如果需求是清楚的，但生成结果偏了，修下游 skill 或生成模板
3. 只修改受影响层
   - 输入匹配问题：优先修脚本文件发现规则
   - 口径问题：优先修 spec 或生成器模板
   - HTML 表现问题：优先修下游 skill 的 assets、脚本模板或渲染逻辑
4. 重新验证
   - 至少复跑对应样例
   - 确认偏差被修正且没有破坏已有能力
5. 回写生成器
   - 若该偏差属于可复用问题，必须同步回写到 `create-report` 的模板、
     校验规则或调整文档，避免下次生成同类错误

### 禁止做法

- 仅在 `report-circle-weekly` 中做一次性手工修补，却不回写生成器规则
- 在没有界定偏差来源前，同时修改 spec、模板、脚本和需求文档
- 把“业务方临时口头反馈”直接写进脚本，而不留文档痕迹
